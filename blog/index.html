Hire-Me Banner

3/29/2017

Try a working example of this widget on jsfiddle here.

This site has been pretty cool to work on and I think its effective in telling my story but...it doesn't do a great job in achieving some of my metrics. Therefore, I'm shutting it down. Its been real guys.

Obviously I'm kidding! I've worked way too hard on this site to just let it go now. Really though, this site could be doing a better job in generating higher CTR. What kind of metrics/CTR would a personal blog need? Why, a contact banner to hire me! So I made one. And now, I'm going to give a quick demo of how you can make one too. A lot of the UX basis for this widget is going to come from the pure CSS nav menu discussed in one of my previous posts. Let's. Get. Started.

The HTML

I love simple HTML. This widget is no different! I embedded my banner in an <aside> tag because (ahem) we're going to cast it aside, now how about that? Ugh. Moving on! We're putting our old friends the checkbox and its label, along with an anchor tag that will use the mailto URL fragment, into this aside element. Here's what that looks like:

<aside class="banner">
<input type="checkbox" id="box"/>
<label id="toggle" for="box"></label>

<a id="collapsible" href="mailto:me@email.com?
subject=Opportunities (From website.com)&body=
Blah blah blah blah blah.%0D%0A
Blah blah blah, blah blah blah blah blah? Here's my contact infomration: %0D%0A%0D%0A

Name: %0D%0A
Phone: %0D%0A
E-Mail: %0D%0A%0D%0A

Thanks,%0D%0A%0D%0A>"
Ready to hire me? Send me an e-mail!
</a>
</aside>

Bam. HTML's done. Simple! I love it. I only wish the CSS were quite so easy...

The CSS

Let's start with the banner itself. This element, for me, is something I want prominently (but not annoyingly) displayed at all times. So I'm going to set it as a fixed position element, somewhere near the top right corner of the page. Out of the way but accessible if someone decides they want to get a hold of me about an opportunity! Inside this <aside> container is the checkbox, its label, and the anchor itself. All four of these need to be styled. You'll notice the styling of the checkbox and its label should be pretty similar to the dropdown widget's style. First, the banner:

.banner {
position: fixed;
top: 100px;
right: 20px;
font-size: 1.1em;
text-align: center;
}

The checkbox graphic is very easy.

#box {
display: none;
}

That one's done. Next, the shared settings for the checkbox label and the anchor link.

#toggle, #collapsible {
display: block;
color: white;
text-decoration: none;
cursor: pointer;
}

And then the individual settings for the checkbox label and anchor link.

#toggle {
float: left;
background-color: rgba(250, 0, 0, 0.7)
padding: 10px 6px 10px 10px;
/*border-bottom-left-radius: 2px;*/
/*border-top-left-radius: 2px;*/
}

#collapsible {
float: right;
background-color: rgba(220, 70, 0, 0.7)
padding: 10px 16px 10px 4px;
/*border-bottom-right-radius: 2px;*/
/*border-top-right-radius: 2px;*/
white-space: nowrap;
transition: padding 1s;
}

Note that I have not set a width for the #collapsible element; this prevents me from animating the full width shrinking effect but I can't stress how much I dislike hard coding those kinds of things. It only leads to site problems and bad user experiences. Anyway, I've also included the nowrap settings for the white-space property. As the #collapsible shrinks the text will get pushed and automatically wrap itself downward, expanding the height of the entire .banner. In turn, this makes the #toggle look funny. Best to just avoid that altogether and set a nowrap!

That wraps up (huehue) the display of all the elements. Now we need to implement their behavior using our friend the checkbox's :checked pseudo-class. I left the label's container devoid of any content so that I could set it using the #toggle's :after pseudo-class. When the checkbox is not checked, the label shows a fitting '/>' close tag that demonstrates that clicking the anchor will collapse the banner. After clicking the '/>' close tag the checkbox gets checked, the collapse "animation" triggers, and the display of the label changes to the '<' open tag. The settings for the #collapsible element are there to round out the transition effects and make sure it looks good.

#box ~ #toggle:after {
content: "/>";
}

#box:checked ~ #toggle:after {
content: "<";
}

#box:checked ~ #collapsible {
width: 0;
padding: 10px 0;
overflow: hidden;
}

And that's it. Another fun, pure CSS widget done!

Master of None

3/29/2017

Coincidence is when you recently right a blog post about not finding any open source projects that stroke your fancy and then a couple of days later Google announces a new site that tracks their Open Source contributions. I can't make this stuff up people, true story. I'm excited to dive in and see what options are available. The Android Open Source Project is something I've been itching to contribute to but in looking at Google's guide to setting up the development environment there is a recommendation to have 100 GB worth of storage space. Unfortunately my development machine does not meet those requirements.

My primary approach to finding a lasting open-source project to contribute to has been to perform some "market research". Try to find sites that track desires for solutions to real problems that people have. This has the benefit of knowing there is a market for that software and that all the time investment put into your endeavor won't be completely unappreciated. Mikael Brevik put together a git repository for the purpose of allowing people to open issues where they could request software solutions or collaborations on different problems. A fitting approach but I'm not totally convinced its reaching a large market of people.

Still, using this approach has left me in the position where I feel like I'm coming into a project late in the game and it would take more time to learn how to contribute, both in getting a deeper understanding of a language and in understanding the project's contributions standards, than to actually contribute myself. Obviously this is part of what it means to be in the development field, to be able to pick up someone else's project, learn it, and help maintain it. But at least if I'm working in that field it means I'm getting paid for it! /s

I think it seems fitting for my next project to be an open source alternative for absence management in the education system. Its a cause I'm personally invested in being a substitute teacher myself and I could write the software using any number of different languages. The current industry standard is AESOP but it requires a paid subscription to a notification service that I consider insulting, not just to me but to any district that pays for their service. It will also likely require a significant amount of UX consideration and as that is my ideal position I think this project would be a perfect match for me.

Jack of All Trades...

3/27/2017

Irony is when you run out of project ideas now that you have a nifty blog where you can share all the cool things you often work on. This is where I have been the past couple of weeks. As such I had turned my attention to the open source horizon. Oh what a promising horizon it is; wrought with untapped potential, endless possibilities, guaranteed fun, a sense of rewarding, and...

...nope. I did not know where to begin. Nothing to fret over though; millions have been in this spot before and no doubt there are plenty of resources to help someone in this position. GitHub's Explore feature was my primary resource as I am well versed in git and have contributed in very small parts to that community. But I would like to find something bigger to work on, something to keep me occupied for a longer duration.

I suppose my problem boils down to not feeling fully comfortable investing myself in a single language yet. I know a moderate amount about a lot of languages but breaking into this industry often requires a specialization, of which I would say I don't have one (yet). My language preference is likely C++ but in my area there aren't many positions that require C++ developers. Python would be a great alternative but it feels like that language market is already well-saturated. So it becomes this balancing game of trying to match a language I would enjoy and would be profitable for me to invest in with an open source project that I can also invest fully in. Turns out, that's much harder than I thought it would be!

Revisiting CSS Toggle DropDowns

3/18/2017

Try a working example of this widget on jsfiddle here.

I worked on my proprietorship's website a couple months ago and got a beautiful, functional, toggle-able, dropdown menu working for navigational purposes. Naturally, that site is hosted from my Raspberry Pi and recently I have not had access to its files. No matter! We can rebuild it; we have the technology.

This navigational menu is to be used on this very site - for the purposes of respecting responsive web design. It needs to be pure CSS (because javascript makes for horrible UX!), work across multiple browsers both desktop and mobile, and of course, it needs to look good. My approach will be to use a dropdown type of menu that employs a clickable toggle. The way this was accomplished with great support for my other website was by using a checkbox input and its clickable label and I expect to borrow that methodology.

So, here is what I was able to come up with. You could of course see for yourself using various browser inspection tools but I'll assume you just want the filtered version. First, the HTML.

The HTML

The checkbox input and its label, from my testing, has served me best when included outside everything (i.e. a child only to the body tag). Here's their extremely simple code:

<input type="checkbox" id="menuCheckBox"/>
<label id="menuBox" for="menuCheckBox"></label>

As you can see, its VERY simplistic as syntax goes and that's what I strive for. Note that actual text can be used if set between the opening and closing <label> tags.

My objective is to be able to create a navigational dropdown menu so elements I want to reveal are stored in the <nav> tag. Within the <nav> tag I use two generic <div> containers to hold navigational links (home, about, etc.) and contact links (GitHub, Facebook, etc.), with a <hr> to visually separate them. Here they are:

<nav>
<div class="pages">
...
</div>
<hr id="navDiv"/>
<div class="contact">
...
</div>
</nav>

Beans. Let's include some actual links in those generic <div> containers to take users around the site. Here's a sample navigational menu with links to pages on the website and external social media sites.

<input type="checkbox" id="menuCheckBox"/>
<label id="menuBox" for="menuCheckBox"></label>

<nav>
<div class="pages">
<a href="index.html" class="active">home</a>
<a href="about.html">about</a>
<a href="blog.html">blog</a>
</div>
<hr id="navDiv"/>
<div class="contact">
<a href="https://github.com/nathanblair" target="_blank">
<img class="contact-icon" src="_res/icons/github.png" alt="X">
</a>
<a href="https://facebook.com/TheGr8Nate" target="_blank">
<img class="contact-icon" src="_res/icons/facebook.png" alt="X">
</a>
<a href="https://linked.com/in/engineerblair/" target="_blank">
<img class="contact-icon" src="_res/icons/linkedin.png" alt="X">
</a>
</div>
</nav>

Cool beans. So that's done. On to the CSS.

The CSS

First, let's put together the checkbox functionality. We want to hide the checkbox graphic but keep its label. In my case, I want the label to be a placeholder with a background image. In that case, here's the css necessary to get the checkbox visuals right:

/*Hide the checkbox graphic*/
#menuCheckBox {
display: none;
}

#menuBox {
display: block;
z-index: 5;
background-image: url('../_res/icons/menu.png');
background-size: 48px;
background-repeat: no-repeat;
background-position-x: center;
background-position-y: center;
height: 50px;
padding: 8px 0;
}

I normally swear against using hard-coded height settings but in this case, since the label element is only serving to display the background image, the height must be set explicitly otherwise the label element would not show up at all. Nominally the background image would be an icon (in my case, the hamburger menu icon), and I've explicitly set the dimensions of the icon to be 48px x 48px. From my experience, this is a manageable size on any browser regardless of desktop or mobile usage.

Next, the settings for the navigation menu itself:

nav {
display: none;
width: 100%;
overflow: hidden;
}

nav a {
text-decoration: none;
display: block;
text-align: center;
}

.pages a {
/*padding: 0.5em 1em;*/
}

.contact a {
display: flex;
justify-content: center;
/*padding: 0.5em 1.3em;*/
}

Now its time to implement the desired behavior for the toggle functionality. This is done using the :checked pseudo-class selector. Since the <nav> element and <input> element are siblings, we can apply all kinds of settings to the <nav> element and its children when the :checked flag is ticked by using the ~ sibling selector.

#menuCheckBox:checked ~ nav {
display: inline-block;
z-index: 5;
position: absolute;
}

#menuCheckBox:checked ~ #menuBox {
background-image: url('../_res/icons/close.png');
}

And thats it. A sample working toggle-able navigation menu that can be used to help keep sites responsive on a wide array of devices and browsers.
